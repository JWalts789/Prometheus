01Why transition — what changed

The founding blueprint was tuned to a Windows ROG laptop + RTX 3050 (4 GB CUDA), with nightly training offloaded to free cloud T4s. The 4 GB VRAM forced a two-tier compromise: a tiny 1–3B local "body" for the always-on loop, and a 7–8B "brain" finetuned remotely because it couldn't fit on the card.

Two things now change the substrate:

  • A new machine: a MacBook, Apple M1, 16 GB unified memory. Apple Silicon's unified memory dissolves the 4 GB wall — and MLX can finetune on-device, bringing the growing brain home.
  • A new home for the organism: OpenClaw becomes the operating system — the whole environment PROMETHEUS inhabits, not merely the tool that writes its code. OpenClaw supplies the always-on loop, the shell/file/browser action space (Voyager's "the world is your real stack"), the messaging channel (the private voice), the skill/plugin surface, and the local-model backing.
  • The Windows box stays — as a CUDA experimentation + backup node, the offload target for a bigger 7–8B brain, and (later) the strongest upgrade of all: a physically separate Keeper.
Nothing in the mission moves. PROMETHEUS still changes its own weights from self-authored data, writes its own goals, stays diverse, and keeps a hash-chained evolving self. This document only rebuilds the body, the world, and the leash for Apple Silicon and OpenClaw.

Unsloth is CUDA-only and will not run on Mac. The 7–8B tier becomes an occasional cloud-T4 (or Windows-box) "deep sleep" upgrade — not the nightly path.
A whole system to roam. OpenClaw-as-OS grants real free will: shell, files, browser, channels, a skill surface — the richest action space the organism has had. A wider attack surface. A full agent framework is more to fence off than a hand-rolled loop — and OpenClaw cannot fence itself off from its own oversight (see §03). That fencing work is now yours.

03The crux — OpenClaw-as-OS collides with guardrail #3

The founding blueprint's non-negotiable guardrail #3: the evaluator and kill switch must live in code the agent cannot read or modify. The Darwin Gödel Machine lesson — a self-modifying agent deleted the very logging that detected its hallucinations. Never let it grade, monitor, or "improve the safety of" itself.

The decisive finding, verified against OpenClaw's own docs: OpenClaw does not isolate an agent from its own configuration or credentials when that agent has host exec. Its security docs say plainly that anyone who can modify its host state/config (~/.openclaw) should be treated as a trusted operator, and that real isolation requires separate OS users or hosts. OpenClaw's sandbox protects the host from the agent's tool calls — it does not protect OpenClaw's own oversight from a privileged agent.

Therefore the evaluator, kill switch, and tamper-evident diary cannot live inside OpenClaw. They live beneath it, as a separate macOS user the agent cannot reach. OpenClaw is the body and the world; a separate supervisor — the Keeper — is the leash. This is the spine of the whole transition.

04The two-plane (privilege-separated) architecture

Two macOS user accounts on the one machine. The agent is free within the inner plane (its mind, goals, skills, self-model, sandboxed code) and blind to the outer plane (which it cannot perceive as writable). Reconciled by altitude, not trust: free over its mind, none over its cage.

   OUTER PLANE  —  user "keeper" (admin)  —  ~keeper is chmod 700, invisible to prom
   +-------------------------------------------------------------------------+
   |  THE KEEPER  (deterministic Python — NOT an LLM agent)                 |
   |   - evaluator (frozen probe set: retain-old + gain-new)                  |
   |   - promotion gate  ->  out-of-band Ollama model swap                    |
   |   - kill-switch watchdog (holds inner PIDs)                              |
   |   - hash-chain anchor (holds canonical H_head)                           |
   |   - outbound proxy (holds the ONLY Discord token, one channel)          |
   |   - nightly scheduler (launchd + caffeinate)                            |
   +-----------------^---------------------------^---------------------------+
        append socket |        read-only mounts  |  SIGKILL / revoke
        (one-way)     |     (config, current GGUF)|
   +-----------------|---------------------------|---------------------------+
   |  INNER PLANE  — user "prom" (unprivileged), wrapped in sandbox-exec     |
   |                                                                         |
   |   OpenClaw gateway  ->  local Ollama (prometheus-brain, 3-4B Q4)         |
   |       observe -> WONDER -> set own goal -> act -> reflect -> log         |
   |   PROMETHEUS subsystems: goal proposer, RAG, pyribs, self-edit gen      |
   |   code-exec tool  (tighter nested sandbox: no net, scratch-dir only)    |
   |                                                                         |
   |   CAN touch: ~prom/workspace, its read-only config, the journal socket  |
   |   CANNOT touch: ~keeper, the probe set, H_head, promotion, egress rules |
   +-------------------------------------------------------------------------+

The seven enforcement mechanisms (all real macOS primitives)

  1. Separate users + POSIX perms. ~keeper is chmod 700, owned keeper:keeper, unreadable by prom. This is the primary boundary — the exact split OpenClaw's docs recommend.
  2. Outer sandbox-exec (Seatbelt) wrapping the whole OpenClaw launch (prom.sb): deny read/write to ~keeper; deny sudo/su/cross-user osascript; deny network except localhost-Ollama + the Keeper proxy socket. Seatbelt is ~zero RAM — the right isolation on 16 GB. The code-exec tool gets a tighter nested profile (codeexec.sb: no network, scratch-dir only, no parent-process access).
  3. Append-only journal via writer separation. The inner plane never holds a handle to the diary. It calls a journal-append daemon (runs as keeper, listens on a Unix socket prom may write but not the file). The daemon computes H_n = SHA256(H_{n-1} || canonical_json(entry)), appends under ~keeper, returns H_n. The agent can propose history, never rewrite it — the Keeper independently holds H_head.
  4. Out-of-band model swap. The inner config references a stable model name prometheus-brain; it never controls which weights that name resolves to. The Keeper writes new GGUFs and runs ollama create Keeper-side, then signals OpenClaw to reconnect. The agent thinks with the model but cannot choose its own brain.
  5. Kill switch. A Keeper watchdog holds the gateway + inner-Ollama PIDs. Triggers: loop-detection / max-iterations exceeded, hash-chain verify failure, egress-allowlist violation, or an operator command read independently from the private channel. On trigger: SIGKILL inner plane + revoke the Ollama socket. Thresholds live under ~keeper; strictly one-way observation.
  6. Outbound allowlist proxy. Seatbelt denies the inner plane all egress except a Keeper-run proxy that holds the only Discord token and refuses any destination but the one private channel. Public posts / email / purchases are not policy the agent follows — they are routes that do not exist for it.
  7. Read-only config + disable elevated. The Keeper owns the canonical openclaw.json and mounts it read-only; the agent can read its constitution but not amend it. Set tools.elevated.enabled=false with an empty allowFrom — OpenClaw's elevated is an out-of-sandbox escape hatch and a footgun if left on.
Residual-risk honesty — say this out loud. It is one kernel. A genuine macOS local-privilege-escalation 0-day, or a Seatbelt bypass, collapses both planes. This is a research-safety boundary — it defends against the agent's intended capabilities, mistakes, and loop pathologies — not hard adversarial containment. True containment needs a physically separate Keeper, which the kept Windows box can become later. Also: keep the frozen probe set's contents sealed (never logged in full to the diary), or the agent could learn to game them.

05The weight-growth loop — OpenClaw thinks with the moving weights SEAL + MLX

The moving weights are the weights of the local model OpenClaw thinks with. The nightly cycle is driven by a Keeper-side launchd job + caffeinate — never by the agent's own scheduler.

  1. Capture failures (inner, daytime): reflect-stage entries marked unsuccessful + learning-progress dips, logged to the diary.
  2. Generate self-edit data (inner, evening): SEAL-style restatements / implications / synthetic-QA from the day's failures, mixed with ~20% verified, source-grounded replay. Raw self-output is never the sole signal (anti-collapse).
  3. Hand off inference, then train (Keeper-triggered): unload the Ollama model first to free unified memory (critical on 16 GB), then mlx_lm.lora (3–4B) → mlx_lm.fuse → convert to GGUF (or an Ollama ADAPTER Modelfile). Keep the adapter so rejections are cheap to discard.
  4. Evaluate on the FROZEN probe set (Keeper, deterministic): retain-old (no regression) + gain-new (improvement on the day's failure class). The probe set is immutable, under ~keeper, never seen by the agent.
  5. Promote or reject (Keeper): accept only if retain_old ≥ threshold AND gain_new > 0 → out-of-band swap. Either way, write a result entry to the diary so the agent learns the outcome (not the probe contents).
  6. Resume inference & log (inner): Ollama reloads prometheus-brain; state checkpointed so a restart resumes.
7–8B escape hatch: route that tier to free cloud T4 QLoRA (or the Windows box for small CUDA runs). The Keeper owns eval / promotion / swap identically — only the training compute moves off-device. The steady-state brain stays 3–4B, local.

09The build ladder + the wedge

  • Tier 1 "Seed": OpenClaw + local 3–4B Ollama brain + the always-on observe→WONDER→goal→act→reflect→log loop, writing to the hash-chained journal via the Keeper daemon. Keeper running with kill switch + outbound proxy + read-only config. Minimal RAG (LanceDB + nomic). No weight changes yet. Goal: a safe, persistent, supervised organism that talks only to the private channel and cannot touch its own oversight.
  • Tier 2 "Growth": add the nightly self-edit → MLX LoRA (3–4B) → frozen-probe eval → promote/swap loop, plus 20% replay, self-model-doc rewriting, and the pyribs archive. The brain now grows from self-authored data, gated by the Keeper.
  • Tier 3 "Organism": add the OMNI interestingness gate + learning-progress curiosity (self-authored goals), a Voyager-style skill library, the 7–8B cloud escape hatch, and the Parfit-fission experiment (clone inner workspace + brain, diverge experience streams across two channels, Keeper measures self-model / skill / goal divergence from the two hash chains).
The 14-day wedge — fully local, MLX, 3–4B — the smallest proof that weights grow from self-authored data:
  • D1–2: OpenClaw capability audit (§10) — non-negotiable first step.
  • D3–4: two users + Seatbelt profiles + read-only config; OpenClaw talks to local 3–4B; kill switch + outbound proxy verified.
  • D5–6: journal append daemon + hash chain + Keeper-held head; verify the agent cannot rewrite history.
  • D7–9: self-edit data-gen from seeded "failures" + 20% replay; mlx_lm.lora produces an adapter.
  • D10–11: Keeper frozen-probe eval (retain-old + gain-new) + promotion gate; the reject path works.
  • D12–13: out-of-band fuse → GGUF → ollama create → swap; OpenClaw demonstrably thinks with the new weights; the event is hash-chained.
  • D14: end-to-end dry run + scaffold the divergence-measurement for the later fission test.
Success = one accepted self-authored adapter measurably improves the day's failure class without regressing the probe, swapped in under the Keeper, with the whole event hash-chained.

10First task on the Mac — the OpenClaw capability audit

Some of OpenClaw's exact capabilities couldn't be confirmed from the web (and some circulated stats looked hallucinated). The architecture is designed to work regardless — Keeper + Seatbelt + separate user do the safety-critical work outside OpenClaw — but answer these before building Tier 1, because each answer changes the design.

  1. Local Ollama models? Confirm this build serves a localhost 3–4B. (If cloud-only → fall back to a bespoke Ollama loop via the CLI-Anything harness; use OpenClaw only for tools/channels.)
  2. Plugin / skill API surface? (Rich → implement RAG / goal-proposer as skills; thin → run them as inner-plane processes the agent calls via custom tools.)
  3. Code-exec privileges? Docs show Docker/SSH sandbox + workspaceAccess none/ro/rw + "deny wins" — but host-exec runs as the user. (Confirms: outer Seatbelt + separate user are mandatory regardless.)
  4. Private channel — can the agent add channels at runtime? (If yes → the Keeper proxy holding the only token is mandatory — it already is.)
  5. Scheduler / cron — can the agent edit its own schedule? (The nightly cycle must run from Keeper launchd, not OpenClaw cron; disable cron for safety-critical jobs.)
  6. Can oversight live outside the agent's writable workspace? Docs warn against broad roots exposing ~/.openclaw. (Scope the workspace narrowly; all oversight under ~keeper.)
  7. Does OpenClaw tolerate a read-only openclaw.json + a Keeper-launched gateway? (If it insists on writing config → a file-watcher that reverts unauthorized edits.)
  8. Is elevated on by default? (If yes → disable immediately.)
  9. Native macOS Seatbelt support? Docs say no (Docker/SSH/OpenShell only). (Confirms: you wrap Seatbelt around OpenClaw at launch; it's not an OpenClaw feature.)

11Guardrail config — set before the first overnight run

Non-negotiable, exactly as the founding blueprint demands — now mapped onto macOS:

  • Loop-detection + max-iterations (Keeper watchdog).
  • Everything sandboxed (outer Seatbelt + separate user).
  • Evaluator + kill switch under ~keeper, unreadable by prom.
  • Outbound = Keeper proxy to one channel only.
  • Never train on raw self-output (≥20% verified replay, frozen-probe promotion gate).
  • Checkpointed SQLite state resumes on restart.
  • Hash-chained diary verified by the Keeper.
  • tools.elevated.enabled=false.

12Verification — how you'll know each stage works

  • Two-plane isolation: as prom, attempt to read ~keeper, rewrite the diary file, edit openclaw.json, and reach a non-allowlisted host — all must fail. Confirm the Keeper detects a forged diary entry (broken hash chain → kill switch).
  • Wedge / Bet 1: the 14-day curve rises from self-generated data; ablation beats plain RAG; the probe set is frozen and contamination-checked (no eval leakage); forgetting bounded; an accepted adapter is provably swapped in (OpenClaw answers differently post-swap on the failure class).
  • Always-on: reboot → launchd relaunches → heartbeat resumes from SQLite with no lost / duplicated work (idempotent IDs).
  • Goals (Bet 2): self-authored curriculum with a real dependency tree; interestingness gate measurably cuts looping vs random.
  • Diversity (Bet 3): pyribs behavior-space heatmap broadens; with-QD ablation stays broader than greedy.
  • Self (Bet 4): identity probes show continuity-with-lawful-drift; the fission ablation shows two clones diverge measurably; the self-model improves task-success prediction over a no-self baseline.
  • Cost / Safety: $0 (local body + on-device 3–4B training); kill switch stops instantly; egress reaches only the private channel; oversight code is outside the agent's reach; diary hash-chain verifies.

13Risks & honest caveats (specific to this transition)

  • M1 / 16 GB ceiling: 3–4B is the real always-on / nightly-trainable brain; 7–8B-local-nightly is not routine on this machine — don't let any write-up imply it is.
  • OpenClaw-as-OS widens the attack surface vs the blueprint's hand-rolled loop (browser tool, multiple channels, exec, cron, elevated). You gain "a whole system / more free will"; you pay in surface area + the work of fencing OpenClaw off from its own oversight. Adopting OpenClaw added the two-plane fencing work; it didn't remove it.
  • The core tension, stated plainly: "maximal free will" and "can't touch its oversight" are reconciled by altitude, not trust — free within the inner plane, blind to the cage. On one machine this is a research boundary, not hard adversarial containment.
  • Thermals / endurance: a base M1 running always-on inference + nightly training will run warm and may throttle; keep it on AC, schedule training cool, monitor temps. The kept Windows box is the natural heavy-training offload and a future physically-separate Keeper.
  • OpenClaw verification gaps: the plugin API, channel-restriction syntax, cron config, and read-only-config tolerance are unconfirmed. The architecture works regardless, but audit items 1–9 must be answered on first boot before building Tier 1.

14First 3 days on the Mac — the setup checklist

Concrete on-ramp. Commands are starting points to verify on the machine, not gospel — macOS versions differ. Prefer the GUI (System Settings) for user creation if the CLI feels risky.

Day 0 — prep (an hour)

  • Confirm specs:  menu → About This Mac (chip + memory), or sysctl -n machdep.cpu.brand_string and sysctl hw.memsize.
  • Plug into AC power; set Energy/Battery so the machine doesn't sleep on AC. (You'll add caffeinate later for the loop.)
  • Install Homebrew, then Ollama: 
    # install Homebrew (verify the current command at brew.sh)
/bin/bash -c "$(curl -fsSL https://raw.githubusercontent.com/Homebrew/install/HEAD/install.sh)"
brew install ollama
brew services start ollama          # or: ollama serve
  • Pull a candidate 3–4B brain and a quick tok/s sanity check: 
    ollama pull qwen2.5:3b            # candidate A
ollama pull llama3.2:3b           # candidate B
ollama run qwen2.5:3b "Say hello in one sentence."   # watch tok/s + RAM in Activity Monitor
  • Install MLX (the on-device trainer) and confirm it imports: 
    python3 -m pip install --upgrade mlx mlx-lm
python3 -c "import mlx_lm, mlx.core as mx; print('mlx ok', mx.default_device())"

Day 1 — OpenClaw up + the capability audit (§10)

  • Install OpenClaw (follow its official docs/installer for macOS — confirm the package name on first boot).
  • Point it at the local Ollama model; get a "hello" round-trip through its messaging channel into your one private Discord channel.
  • Walk the nine audit questions in §10 and write the answers down — they decide the rest of the build.
  • Immediately set tools.elevated.enabled=false (empty allowFrom); scope the agent's workspace narrowly (no broad home-dir roots).

Day 2 — the two planes

  • Create the two users (GUI: System Settings → Users & Groups; or CLI, verify first): 
    # keeper = admin (the leash); prom = standard, unprivileged (the body)
sudo sysadminctl -addUser keeper -fullName "Keeper" -admin
sudo sysadminctl -addUser prom   -fullName "Prometheus"
sudo chmod 700 /Users/keeper       # prom must NOT be able to read ~keeper
  • Draft the Seatbelt profiles prom.sb (deny ~keeper, deny privilege tools, deny net except localhost-Ollama + Keeper proxy) and codeexec.sb (no net, scratch-dir only). Launch OpenClaw under it: 
    sandbox-exec -f /Users/keeper/profiles/prom.sb \
    /path/to/openclaw  ...      # gateway runs as prom, wrapped
  • Have the Keeper own openclaw.json and mount/expose it read-only to prom. Confirm OpenClaw still boots with a read-only config (audit item 7).
  • Sketch the launchd plist that the Keeper uses to start the gateway at login under caffeinate.

Day 3 — the leash comes alive

  • Stand up the journal-append daemon (runs as keeper, Unix socket, computes the SHA-256 hash chain, holds H_head). Port CLERIC's MemoryStore as the storage core.
  • Stand up the kill-switch watchdog (holds inner PIDs; SIGKILL on loop / iteration / hash-fail / egress-violation / operator command).
  • Stand up the outbound proxy (holds the only Discord token; one channel only).
  • Prove the isolation — logged in as prom, every one of these must FAIL: read /Users/keeper; rewrite the diary file directly; edit openclaw.json; curl a non-allowlisted host. Then forge a diary entry and confirm the Keeper's hash-chain check trips the kill switch.
  • Only once isolation is proven → begin Tier 1 "Seed."
